# Plan: Nexus 设备授权风控增强（设备码/CLI 登录）

🎯 目标
- 强化设备码授权的安全性与可追溯性，避免跨网络/异常设备的长期授权滥用。
- 登录历史与设备管理能清晰识别授权来源（App / CLI / External），可随时撤回。
- 保持流程简单、可解释，避免过度打断正常用户。

🧩 范围
- 设备码授权链路（/api/app-auth/device/*）
- 登录历史与设备管理（/api/login-history、/api/devices）
- Dashboard 展示与操作

🚫 非目标
- 不引入外部风控引擎或第三方风险评分服务。
- 不做复杂的人机验证链路（如 Turnstile 二次挑战）。

## 阶段划分

### Phase 0（已落地，待验收）
1. 记录设备码请求的 request_ip / client_type。
2. 授权确认时校验 CLI 发起 IP 与浏览器当前 IP 一致性。
3. 长期授权仅在常用设备 + 常用登录地组合下允许。
4. 设备授权页禁用长期授权并给出原因提示。
5. 登录历史 / 设备管理显示授权来源（App/CLI/External）。
6. 授权完成后执行多策略关闭标签页（window.open + window.close）。

### Phase 1（计划落地）
1. **速率限制**：按 user_id / device_id / IP 限制设备码申请频率。
2. **异常冷却**：连续失败/取消超过阈值时进入冷却期（例如 10 分钟）。
3. **授权审计**：记录授权批准/拒绝/撤销的审计日志（时间、原因、来源）。
4. **可信设备管理**：允许用户标记常用设备/常用登录地（或显式白名单）。
5. **设备授权时间窗**：长期授权需在登录后 N 分钟内完成验证。

### Phase 2（可选增强）
1. **IP/ASN 风险评分**：识别 TOR/代理网络（仅本地规则，不接外部服务）。
2. **异地提醒**：授权成功后邮件/站内提醒（带撤销入口）。
3. **设备指纹校验**：客户端提供稳定 device fingerprint（与 device_id 组合）。

## 验收标准
- CLI 设备码登录成功后，登录历史与设备管理展示来源为 CLI。
- IP 不一致时拒绝授权，并提示明确原因。
- 非常用设备/地点无法选择长期授权（按钮不可用 + 提示）。
- 撤销设备后该设备的 token 立即失效（验证 token_version）。

## 风险与回滚
- 风险：误判导致长期授权入口不可用。
- 回滚：保留短期授权通道；可通过配置开关禁用严格模式。
