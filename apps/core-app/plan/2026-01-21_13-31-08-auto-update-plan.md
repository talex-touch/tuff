# Plan: 自动更新方案梳理与落地

🎯 任务概述
目标是梳理现有自动更新实现与发布配置的偏差，明确统一的更新来源、下载/安装链路与验证策略，
并形成可执行的落地步骤与回归清单，确保多平台更新流程稳定可追踪。

📋 执行计划
1. 现状盘点：整理主进程更新模块、前端设置页、下载中心与发布配置的现状与差异点，输出清单。
2. 更新源策略：确认更新来源（GitHub Releases vs 自建 publish）与渠道策略，形成单一配置口径。
3. 资产规范：定义平台/架构/签名/校验文件命名与必备产物清单，对齐下载与校验逻辑。
4. 流程收敛：统一更新检查 → 下载 → 安装触发路径，明确是否仅走主进程 UpdateSystem。
5. 交互与通知：梳理 UI 展示、提醒策略、忽略/稍后逻辑与下载完成提示，确保一致性。
6. 缓存与频率：确认缓存 TTL、频控策略、重试与错误分类提示，避免频繁请求与误报。
7. 验证与回归：制定跨平台回归清单（渠道切换、断网、限频、签名失败），补必要单测/手测步骤。
8. 交付物整理：输出最终方案文档与配置调整清单，标注依赖与上线前检查项。

## 现状盘点与差异清单
- UpdateService（`src/main/modules/update/UpdateService.ts`）：主进程更新编排与设置/缓存中心；直接访问 GitHub Releases API（含 ETag/TTL/限频/重试策略），持久化 UpdateRepository 与 release cache；内部实例化 UpdateSystem 承担下载/安装，并通过 UpdateEvents 与渲染进程交互。
- UpdateSystem（`src/main/modules/update/update-system.ts`）：独立更新检查/下载/安装实现，内置 GitHub Releases 拉取与平台资产匹配；配置口径为 `autoDownload/autoCheck/checkFrequency/updateChannel`，未读取 UpdateService 的 source/缓存；通过 DownloadCenter 发起高优先级下载并触发通知。
- SettingUpdate（`src/renderer/src/views/base/settings/SettingUpdate.vue`）：前端设置入口；频率选项为 `everyday/1day/3day/7day/1month/never`，与 UpdateSystem `startup/daily/weekly/never` 口径不一致；使用 `GithubUpdateProvider` 展示资产列表，并允许通过 `useDownloadCenter` 直接发起下载任务。
- DownloadCenter（`src/main/modules/download/download-center.ts`）：通用下载模块，更新下载依赖 `DownloadModule.APP_UPDATE` 与通知服务；既被 UpdateSystem 调用，也被 SettingUpdate 直连添加下载任务，存在多入口触发的现实路径。
- 发布配置（`electron-builder.yml`、`dev-app-update.yml`）：`publish` 使用 generic provider 与固定 URL（dev 使用示例 URL）；与 UpdateService/UpdateSystem 直接访问 GitHub Releases 的更新源口径不一致，且渠道映射未在发布配置中显式表达。

⚠️ 风险与注意事项
- 更新源与发布管线不一致会导致无法拉取/下载，需与发布流程协同确认。
- 资产命名或校验策略不一致会导致“无匹配资产”或校验失败，需跨平台验证。
- UI/通知链路分叉可能导致状态不同步，需要统一事件来源与状态存储。
