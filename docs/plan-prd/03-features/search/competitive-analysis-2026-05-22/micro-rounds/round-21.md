- 结论：
  - 主文档判断成立。Tuff 当前并不是“没有 Translate 能力”：`touch-translation` 支持 typed text、多 provider 文本翻译、图片 input 的 OCR / image translate；OmniPanel 有 selected text AI translate；CoreBox 也有剪贴板图片 / image item 翻译路径。
  - 但这些路径仍是分散的：插件命令、OmniPanel 内置 AI action、CoreBox image action 分别维护输入来源、权限、失败码和结果展示。把 Raycast Translate 映射为 `ContextActionProvider` + evidence 是正确边界，不应继续只增强一个孤立插件页面。
  - 当前源码已经支撑首版合同的关键字段：`TuffQuery.text` / `inputs.image`、`acceptedInputTypes`、OmniPanel selection diagnostics、provider-level `status/error/provider/model/traceId`、CoreBox `IMAGE_UNAVAILABLE` / `SCENE_UNAVAILABLE`。下一步缺的是把这些字段统一成可验收 payload，而不是重新设计翻译引擎。
  - 选中文本隐私边界没有被主文档低估：OmniPanel AI metadata 单测明确不记录 raw clipboard / selection 内容；主文档也要求只在用户主动触发时读取，并在 evidence 中记录 source/length/hash 这类脱敏信息。
  - KISS/YAGNI 角度看，首版应只收敛三类样本：typed text 翻译、selected text 翻译、clipboard image 翻译。不要直接做 Raycast 式自然语言 inline translation 全量识别，也不要复制新的翻译面板或后台常驻 selected text 捕获。
- 是否发现需修正的主文档问题：否。`02`、`07`、`10`、`11` 对 Translate 的描述与 live source 一致：能力底座存在，但还缺统一 input source、Context Action taxonomy、provider/failure evidence 和 packaged 验收；没有把 `touch-translation` 夸大成完整 Raycast Translate。
