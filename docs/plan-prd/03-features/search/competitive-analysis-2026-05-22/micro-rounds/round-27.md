# 微审计 27/70

- 审计主题：Alfred Workflow 是否被误解为 Tuff 必须先做完整可视化画布；核对主文档是否把路线收敛为 manifest / Prelude / typed SDK / validate / run record 的小合同，而不是提前建设大而全节点编辑器。
- 读取/核对的文档或源码锚点：
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/03-alfred-workflow-model.md`：第 1 节明确缺口是“统一 workflow contract”，不是缺大型可视化编辑器；第 5 节给出 `workflows[]` contract v1 草案；第 6 节把完整可视化节点画布、任意 Run Script、复杂循环/并行等列为不照搬边界；第 7 节把落地切片拆成 docs / validate / trigger / context-action / debug / Nexus card。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/10-execution-roadmap-synthesis.md`：第 1 节把 Alfred Workflow 价值定义为“先落合同，不落大画布”；第 4.2 节的 `workflow-contract-v1-doc-validate` 只要求 schema、`tuff validate --strict`、dry-run 风险摘要；第 6 节继续禁止照搬 Alfred 完整画布和任意 Run Script。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/11-100-round-cross-review-ledger.md`：第 27 条结论为“Alfred Workflow 是否被误解为必须上画布：否”，并指向 `03`、`10` 与 KB，说明该判断已有台账锚点。
  - `.codexpotter/kb/alfred-workflow-model-2026-05-22.md`：记录 Alfred Workflow 应被理解为 object-flow 模型；Tuff safer path 是 manifest + Prelude + typed SDK contract，并提醒不要把当前 Intelligence Workflow 当成普通 plugin workflow 的完整替代。
  - `packages/utils/plugin/index.ts`：`IPluginFeature` 已有 `commands`、`acceptedInputTypes`、`interaction`、`omniTransfer` 等声明面；这能承接 command / context input 的一部分，但没有 workflow-level `triggers/input/configSchema/steps/output`。
  - `apps/core-app/src/main/modules/plugin/adapters/plugin-features-adapter.ts`：Plugin feature 搜索会按 `acceptedInputTypes` 过滤非文本输入，并能进入 push / webcontent 路径；这证明 Tuff 有 Script Filter / Universal Actions 的底座，但仍是 feature 级执行，不是多步骤 workflow run。
  - `apps/core-app/src/main/modules/plugin/plugin.ts`：`triggerFeature()` 调用 Prelude `onFeatureTriggered(feature.id, query, feature, signal)`，并记录 feature 执行 breadcrumb；这能支撑动态候选和取消，但没有统一 step timeline、变量命名空间或 permission preflight 记录。
  - `packages/utils/types/intelligence.ts` 与 `apps/core-app/src/main/modules/ai/intelligence-workflow-service.ts`：已有 `WorkflowDefinition`、`WorkflowRunRecord`、triggers、steps、history 等 AI workflow 基础；但它偏 Intelligence / model / tool runtime，不应直接等同于通用插件 workflow 合同。
  - `apps/core-app/src/main/utils/workflow-debug.ts`：存在 `.workflow/.debug` 的 best-effort 调试日志写入，能作为 debugger 方向证据；但它不是用户可见的通用 workflow run timeline。
- 结论：
  - 主文档没有把 Alfred Workflow 简化成“先做画布”。`03` 与 `10` 的共同路线是先定义 workflow contract v1，把输入、参数、步骤、输出、失败、权限、调试、发布 metadata 固化成可验证合同。
  - Tuff 当前已有足够底座支持这个低风险路线：Manifest feature 声明能表达 commands / input types / permissions；Prelude `onFeatureTriggered` 能动态返回或推送结果；PluginFeaturesAdapter 能按上下文输入过滤；Intelligence Workflow 与 workflow-debug 提供 run record / step trace / debug 的可复用经验。
  - 但这些底座仍然分散：Plugin feature 是 feature 级，不是 workflow 级；Intelligence Workflow 偏 AI runtime，不是通用插件动作编排；workflow-debug 只是开发期日志，不是完整 debugger UI。因此文档继续把“统一 workflow contract + CLI validate + run timeline + Nexus card”列为后续任务是准确的。
  - KISS / YAGNI 角度看，先做 JSON/manifest contract 与校验，比先做拖拽画布更符合当前项目阶段。画布会放大 schema、权限、调试、发布、迁移、跨平台 shell 安全等未闭环问题；当前最小闭环应先让一条 workflow 能被声明、校验、运行、失败、审计和展示。
  - 安全边界也保持一致：文档没有建议开放 Alfred `Run Script` 等价的任意脚本能力，而是要求 shell 走高风险 `system.shell`、fail-closed、timeout、cwd 校验、输出截断和日志脱敏。
- 是否发现需修正的主文档问题：否。`03`、`10`、`11` 对 Alfred Workflow 的映射是保守且一致的：承认 Tuff 有 Manifest / Prelude / Flow / DivisionBox / Intelligence / tuff-cli / Nexus 的底座，同时没有把这些底座夸大成完整 workflow editor 或 Alfred parity。
