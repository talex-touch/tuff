# 微审计 26/70

- 审计主题：Raycast Windows 核心能力覆盖是否在 Tuff 文档中被完整纳入，并且多输入能力是否能由 Tuff 当前 `TuffQuery.inputs` / `acceptedInputTypes` 事实支撑，而不是只停留在能力清单枚举。
- 读取/核对的文档或源码锚点：
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/01-basic-capability-alignment.md`：第 2.1 节用于核对 Raycast Windows 侧 App Launcher、File Search、Clipboard、Snippets、Quicklinks、Calculator、Emoji、Shortcuts、AI 等基础能力是否进入总矩阵。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/02-raycast-smooth-interactions.md`：用于核对 Search Bar、Action Panel、Quicklinks、Snippets、Dynamic Placeholders、Translate、AI Commands 的映射是否被拆成 Tuff 可落地的合同与 evidence，而不是只列功能名。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/05-search-performance-ranking.md`：第 2.3 节和第 4 节把 uTools 式多输入智能匹配、Raycast Action Panel、Alfred Universal Actions 统一纳入 Context Actions / Action Panel evidence，覆盖文本、图片、文件和 HTML 输入。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/06-parameter-filling-dynamic-variables.md`：第 3 节明确 `TuffQuery.text` / `TuffQuery.inputs`、`acceptedInputTypes`、Snippets placeholders、Translation 输入、OmniPanel selected text、Flow payload 等当前证据和缺口。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/11-100-round-cross-review-ledger.md`：第 26 条判断 Raycast Windows 核心能力已纳入矩阵；第 29、38、44 条把 Universal Actions / 超级面板 / 多数据源触发落到 `acceptedInputTypes` 与 `TuffQuery.inputs`，并标注截图仍需 evidence。
  - `packages/utils/core-box/tuff/tuff-dsl.ts`：`TuffInputType` 定义 `text`、`image`、`files`、`html`；`TuffQueryInput` 定义 `content`、`rawContent`、`thumbnail`、`metadata`；`TuffQuery` 明确区分用户主动输入的 `text` 与剪贴板/附加来源的 `inputs`。
  - `packages/utils/plugin/index.ts`：`IPluginFeature.acceptedInputTypes` 声明插件 feature 可接收的输入类型；注释说明未声明时默认兼容文本输入，且 `onFeatureTriggered` 可接收 string 或完整 `TuffQuery`。
  - `apps/core-app/src/main/modules/plugin/adapters/plugin-features-adapter.ts`：搜索插件 feature 时会读取 `query.inputs`，对非 text/html 输入要求 feature 显式声明 `acceptedInputTypes` 并全部接受，避免图片或文件误进 text-only feature。
  - `apps/core-app/src/renderer/src/modules/box/adapter/hooks/clipboard-query-inputs.ts` 与 `useClipboardState.ts`：把剪贴板图片、文件、HTML、长文本或显式文件模式转换为 `TuffQueryInput[]`，说明多输入不是文档想象，而是 CoreBox query 构造路径的一部分。
  - `plugins/touch-translation/manifest.json` 与 `plugins/clipboard-history/manifest.json`：Translation 目前主要声明 text 输入；Clipboard History 声明 text/image/files/html，说明官方插件已开始按输入类型收口能力边界。
- 结论：
  - 主文档第 26 条“Raycast Windows 核心能力是否覆盖全面”的结论成立。Raycast Windows 能力清单没有只停在 App Launcher / File Search 两项，Clipboard、Snippets、Quicklinks、Calculator、Emoji、Shortcuts、AI 和 Action Panel 相关能力都已进入专题矩阵或后续合同。
  - Tuff 当前源码已经具备承接多输入核心能力的底座：`TuffQuery.text` 与 `inputs` 语义分离，`TuffInputType` 覆盖 text/image/files/html，CoreBox 前端能从剪贴板和文件模式构造输入，插件 adapter 会按 `acceptedInputTypes` 过滤非文本输入。
  - 但这不等于 Raycast Windows 或 uTools 超级面板已经完整对齐。当前缺口仍是用户可见 evidence：输入来源、匹配原因、provider health、OCR/截图失败原因、Action Panel 可执行动作数量与失败态还没有统一展示；截图输入也应继续保留为需验证项，不应宣称完成。
  - 下一步应延续主文档的低风险路线：把 selected text / clipboard image / clipboard files / HTML text 四类样本接入 Context Actions evidence，记录 source、privacy、matched feature、accepted input type、failure reason；不要先重写搜索框或复制 uTools 鼠标超级面板。
- 是否发现需修正的主文档问题：否。`01`、`02`、`05`、`06`、`11` 对当前 Tuff 能力的表述整体谨慎：既确认了多输入和插件声明底座，也明确截图、source evidence、failure reason、ContextActionProvider 仍是后续工作，没有把现状夸大为完整竞品同款。
