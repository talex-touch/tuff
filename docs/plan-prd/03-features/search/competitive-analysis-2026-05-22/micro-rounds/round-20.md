# 微审计 20/70

- 审计主题：Snippets Dynamic Placeholders 是否已与 Raycast Snippets 对齐，还是当前 `touch-snippets` 只完成了有限的 legacy placeholder 闭环，后续应优先抽统一变量 resolver，而不是继续在插件内扩硬编码替换。
- 读取/核对的文档或源码锚点：
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/01-basic-capability-alignment.md`
    - Snippets 被标为“已落地但仍需产品化”，当前证据包括 `touch-snippets` 的 text/code/prompt/template、`{{date}}` / `{{time}}` / `{{uuid}}` / `{{clipboard}}`、CloudShare pack 和敏感内容过滤。
    - 同文明确 Raycast Snippets 还有 tags、keyword auto-expansion、`{cursor}`、`{random}`、`{calculator}` 等动态占位符，因此 Tuff 不能把四个 placeholder 宣称为完整对齐。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/02-raycast-smooth-interactions.md`
    - 第 2.3 节把 Raycast 参数填充拆到 Quicklinks、Snippets、AI Commands、Command Arguments 四条线，要求后续用 `TuffParameterSet` 统一。
    - 文档建议 `touch-snippets` 保持旧语法兼容，再映射到同一 resolver，而不是复制一套新模板引擎。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/06-parameter-filling-dynamic-variables.md`
    - 把 `touch-snippets` 当前能力明确限定为 `{{date}}` / `{{time}}` / `{{uuid}}` / `{{clipboard}}`。
    - 指出当前 placeholder 由插件内硬编码替换，`clipboard` 无内容或无权限时容易被替换为空，缺统一 failure reason。
    - `snippets-placeholder-v1` 验收建议要求无权限时显示 `PERMISSION_MISSING`，而不是静默成功。
  - `docs/plan-prd/03-features/search/competitive-analysis-2026-05-22/11-100-round-cross-review-ledger.md`
    - 第 20 条已记录：Snippets Dynamic Placeholders 差距明确，文档没有把首版 placeholder 宣称完整。
  - `plugins/touch-snippets/manifest.json`
    - 插件声明 `sdkapi: 260428`，`clipboard.write` 为 required，`clipboard.read` 为 optional；三个 feature 都只声明 `acceptedInputTypes: ["text"]`。
    - `permissionReasons.clipboard.read` 明确用于 `{{clipboard}}` 占位符与从剪贴板保存片段。
  - `plugins/touch-snippets/index.js`
    - `applyPlaceholders()` 只替换 `{{date}}`、`{{time}}`、`{{uuid}}`、`{{clipboard}}`。
    - `readClipboardTextForContent()` 只有内容包含 `{{clipboard}}` 时才调用 `readClipboardText()`，符合最小授权原则。
    - `readClipboardText()` 在缺少权限或读取异常时返回空字符串；`onItemAction()` 随后仍把 resolved 文本写入剪贴板，没有把缺剪贴板输入显式转成失败态。
    - `createSnippetPack()` 默认过滤敏感片段，说明 Snippet 分享包已有隐私保护切片，但这与 placeholder resolver 的 failure/evidence 不是同一层能力。
  - `plugins/touch-snippets/index.test.cjs`
    - `applyPlaceholders resolves date time uuid and clipboard placeholders` 只覆盖 date/time/uuid/clipboard 四类替换，并验证同一次替换中 `uuidFactory` 只调用一次。
    - `createSnippetPack filters sensitive snippets by default` 覆盖敏感内容过滤，支撑“分享包隐私边界存在，但 Dynamic Placeholder 仍需 evidence”的区分。
- 结论：
  - 主文档口径成立。`touch-snippets` 不是空壳：它已经能搜索、保存、复制文本/代码/提示词/模板片段，并在复制时解析四个首批 placeholder。
  - 当前 Dynamic Placeholders 只能算 Tuff Snippets V1 的有限闭环。它没有 `{cursor}`、`{calculator}`、`{selection}`、browser tab、named argument、modifiers、默认值、字段校验或 placeholder source evidence。
  - `{{clipboard}}` 的隐私处理方向是对的：只有实际包含该占位符时才申请 `clipboard.read`。但失败语义还不够诚实：无权限、空剪贴板或读取异常都会被折叠为空字符串，用户看不到 `PERMISSION_MISSING` / `SOURCE_UNAVAILABLE` 这类 reason。
  - 因此下一步不应在 `touch-snippets/index.js` 里继续堆更多 `replace()`。更小且可复用的路径是先抽薄的变量 resolver：兼容 `{{date}}` 等旧语法，同时把 clipboard/selection/calculator 等 source、privacy、failure reason 和 evidence 统一给 Quicklinks、Snippets、Translate、AI Commands 使用。
  - KISS/YAGNI 角度看，系统级 hot string、自动粘贴、cursor 精确回填和 browser-tab 变量都应后置；首批只需要保证旧 placeholder 不破坏、失败不伪装成功、resolver 可被 Quicklinks/AI Commands 复用。
- 是否发现需修正的主文档问题：否。`01`、`02`、`06`、`11` 都没有把 `touch-snippets` 夸大成完整 Raycast Snippets / Dynamic Placeholders 对齐；它们把当前四个 placeholder 写成已落地基础，把 cursor/calculator/selection/evidence/统一 resolver 写成后续缺口，与源码一致。
