迁移波次与决策门

推荐实施波次

Wave 0
每个候选先做 contract inventory 与 focused tests；记录调用图、行为、视觉/a11y parity、package exports 与 platform matrix；不移动源代码。
Wave 1
低风险本地简化与确认的 app-local duplicates：CR-01/02/04/05/10/11/12、NX-09/10、PL-01；不改公共 package。
Wave 2
Tuffex parity migrations：先 TButton/TuffIcon thin adapters，再 TTag/TCheckBox；tabs/progress/plugin controls 要明确 parity 工作后才处理。所有调用方迁移与 legacy 删除必须在同一 green phase。
Wave 3
utils consolidation：canonical sleep、translation pure data URL helper；推迟 music utils 以及 hash/signing generalization。
Wave 4
typed OSAdapter/domain capabilities：wallpaper、app launch、AI clipboard/openExternal；每阶段一个 capability，并做 win/darwin/linux tests。
Wave 5
large monolith splits：一次一个 domain，治理、providers、plugin module、quick-ops、search、update；禁止 mass rewrite。
Wave 6
功能波次之后才做 cleanup/naming：清理选定 ignored outputs、决定 tracked .bak、可选地单独规划 package/plugin rename。

  3. DG-03 Tracked backup：若仍有效，推荐将 examples/tuff-builder.example.ts 改为 formal example；删除必须明确批准。
