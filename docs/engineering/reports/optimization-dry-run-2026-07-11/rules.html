Interpretation protocol

规范性用语、严重度与置信度

本手册的用语解释
用语执行含义
MUST / 必须没有例外的前置条件。条件未满足，迁移不得开始或不得合并。
SHOULD / 应当默认方向。若偏离，变更提案必须记录具体理由、替代措施与剩余风险。
MUST NOT / 不得禁止的实施方式。它不能以“方便”“行数多”或“语法相似”作为例外理由。

严重度

  • Critical：存在数据、安全或 API 破坏风险，必须立即阻止。本审计未确认 Critical。
  • High：重复行为或混合职责，且边界清晰；迁移会触及多个调用方或平台契约。
  • Medium：可信的整合候选，但仍需 parity、调用图或 runtime 证明。
  • Low：局部简化或观察，不得被扩展成架构项目。

置信度的受限含义

置信度只表示已收集证据对某项建议的支撑程度，不表示行为、视觉或跨平台 parity 已被证明。简报要求报告 confidence，但没有定义数值或高/中/低标签。

显式歧义 在未另行批准量表前，不得把“高置信度”解释为可直接迁移，也不得把任何置信度解释为删除授权。缺失的 runtime graph、视觉套件、平台矩阵与行为测试仍然是阻塞事实。

R1

Renderer/Vue 提取规则

Vue SFC 应当保留渲染、props/emits 与视图专属交互。包含多个 computed/watch/watchEffect/lifecycle/event/transport 职责的连贯 state/effect/orchestration，应当移动到 composable 或 app-local module。

提取资格决策树

  1. 是否存在可命名的单一职责，而非仅仅“文件很长”？否：不得提取
  2. 该职责能否定义稳定的输入/输出契约？否：保留本地并降低耦合
  3. 是否有可观察的测试 seam？否：先建立验证 seam，不迁移
  4. 抽出后是否保留渲染、props/emits、视图交互在 SFC？否：重新划分边界
  5. 候选是否携带 app state 或 SDK/transport coupling？是：保持 app-local
  6. 候选 composable 是否仅因语法类似而被合并？是：不得合并
  7. 语义、lifecycle ownership、error behavior 是否都一致？是：可作为受证据约束的提取候选
  • MUST：只提取具名职责、稳定 input/output contract 与可观察测试 seam。
  • MUST：只有语义、lifecycle ownership 和 error behavior 都相同，才能合并相似 composable。
  • MUST NOT：只按行数提取，或仅因语法相似合并。
  • MUST NOT：为缩短文件而把 app-specific state、SDK 或 transport coupling 升格到公共包。

审计解释：TuffUserInfo.vue 的身份/账户网络/格式化/对话框动作混杂约 330 script 行，但只允许先处理 CR-01/CR-02，拒绝任意进一步切分。FlipDialog.vue 的 visibility/focus 候选不得在行为 parity 前与通用 dialog focus 合并。FlatCompletion.vue 的两个 watcher 可合并为 watch([value,res], ...)，无需 composable；AgreementTemplate.vue 与 TTabItem.vue 分别适用 computed/direct prop 替换，而不是抽象化。

R2

Tuffex 组件整合规则

  • SHOULD：若 Tuffex 已有行为等价 primitive，消费者应直接委派给 Tuffex。
  • MUST：若本地组件是可复用生态 primitive 且 Tuffex 缺失，迁移 implementation/API/styles/tests/export 到 packages/tuffex，迁移每个 caller，然后删除 legacy local file。
  • MUST NOT：内部 clean cutover 后保留 wrapper、shim、alias、compatibility-only helper 或 duplicate export。
  • MUST NOT：仅因组件渲染 UI 就把 domain component 移出 app/plugin。它可组合 Tuffex primitive，仍应留在所属域。

End-state contract

清洁切换与 legacy 删除契约

内部 clean cutover 的完成定义 当且仅当每个内部 caller 已迁移、目标 API parity 已证明、所有 required evidence 已记录后，legacy implementation 才必须在同一 green phase 删除。完成不等于“新旧同时可用”。
  1. 先盘点 caller、导入、exports、registries、manifests、runtime IDs 与测试 seam。
  2. 完成目标实现或直接委派，并证明 API、keyboard/a11y、event、slot、CSS/theme injection、Vue peer、bundle、visual parity。
  3. 迁移所有内部 caller，并验证没有 legacy call graph。
  4. 删除 legacy file、obsolete code、alias、deprecated wrapper、compatibility-only helper 与 duplicate export。
  5. 仅 public Tuffex/utils export 适用 DG-01 的 additive 默认，不得把它误用于内部 app 文件。
公开兼容性的边界 简报只授权“内部 app files clean cutover”和“public Tuffex/utils exports additive unless major-version breaking change approved”的默认。任何特定 export 是否公开、是否允许 breaking change，仍需 DG-01 明确批准，不能在本页自行判断。

Sequencing

迁移波次纲要

Wave 0对每个 candidate 做 contract inventory 与 focused tests，不移动源。捕获 call graphs、behavior、visual/a11y parity、package exports、platform matrix。
Wave 1低风险 local simplification 与已确认 app-local duplicate composables：CR-01/02/04/05/10/11/12、NX-09/10、PL-01。不得改变 public package。
Wave 2Tuffex parity migrations：先 TButton/TuffIcon thin adapters，再 TTag/TCheckBox；tabs/progress/plugin controls 必须先完成明确 parity 工作。所有 caller 在同一 green phase 迁移，且删除 legacy。
Wave 3utils consolidation：canonical sleep、translation pure data URL helper；延期 music utils 与 hash/signing generalization。
Wave 4typed OSAdapter/domain capabilities：wallpaper、app launch、AI clipboard/openExternal。每个 capability 单独一个 phase，并有 win/darwin/linux tests。
Wave 5大型 monolith 按一个 domain 一次拆分：governance、providers、plugin module、quick-ops、search、update。不得 mass rewrite。
Wave 6功能波次后再处理 cleanup/naming：选择性清除 ignored output、决定 tracked .bak、另行计划 package/plugin rename。

Recommended defaults

评审与实施推荐默认值

  • 默认应当从 Wave 0 的 contract inventory 开始，而不是从代码移动开始。
  • 默认应当优先 local composable 与 local pure engine，直到存在稳定第二 consumer 或 foundational domain ownership。
  • 默认应当先处理低风险 local simplification、已确认 app-local duplication 与 thin adapter parity，再处理 public package 或 platform capability。
  • 默认应当对 public export 采取 additive 策略，对内部 app file 采取完整 clean cutover。
  • 默认应当把 generated/rebuildable ignored outputs 作为确认后的 cleanup，不把它们当 source defect。
  • 默认MUST NOT把名称不一致、文件很长、组件外形相近或出现 process.platform 作为自动迁移/删除理由。
  • 默认MUST在每个 destructive、public compatibility、runtime naming 与 platform 行为变更前，记录适用 DG、证据、限制与批准结论。
